D. Shift + Esc

看到数据量较小（n,m≤200n,m\le 200n,m≤200），那么思考暴力 DP 可不可行。

假设我们在 (i,j)(i,j)(i,j)，如果要移动到下一行的 (i+1,j)(i+1,j)(i+1,j) 就需要考虑下一行平移了多少，才能知道在 (i+1,j)(i+1,j)(i+1,j) 这个格子上的数是哪一个；如果是向右移动到 (i,j+1)(i,j+1)(i,j+1)，那么也需要考虑这一行移动了多少，才能知道 (i,j+1)(i,j+1)(i,j+1) 格子上的数是哪一个。

所以我们发现无论如何都需要枚举平移了多少这个量。那么我们的 DP 策略就是按行计算 DP

  1. 枚举当前行平移多少距离 sss
  2. 计算：如果平移 sss，那么这一行的 DP 值为多少（从上一行转移过来）
  3. 计算：如果平移 sss，那么在这一行向右走能够更新的 DP 值（从左侧转移过来）

这里枚举完行数之后，必须先枚举平移距离。如果先枚举纵座标的话，就必须再额外保存“当前行平移距离”这个信息（也就是 dp\tt dpdp 数组需要开三维），才能保证在计算同一行时，平移距离是相同的

dp[i][j][s]→dp[i][j+1][s]\mathtt{dp}[i][j][s]\to\mathtt{dp}[i][j+1][s]dp[i][j][s]→dp[i][j+1][s]

总时间复杂度 O(nm2)O(nm^2)O(nm2)

E. Broken Queries

能注意到的一点是，33=O(log⁡n)33=O(\log n)33=O(logn)，所以我们的算法应该在 log⁡\loglog 级别。

log⁡\loglog 级别的算法，我们可以想到二分，那么我们对什么东西进行二分呢？

一开始的想法是二分出 111 的位置。但这和我们的答案几乎无关（很难通过 111 的位置反推 kkk）

所以我们应该二分 kkk，然后注意到这样一个性质：

  1. 如果两段区间都是全 000，那么无论是否两者长度都 ≥k\ge k≥k，他们的回答都应该一致
  2. 否则如果一段含 111，另一段不含，那么他们的回答一定不同。

所以，我们可以询问 (1,n4),(n4+1,n2)(1,\frac n4),(\frac n4+1,\frac n2)(1,4n​),(4n​+1,2n​)，得知 111 的位置，然后再询问 (1,n2)(1,\frac n2)(1,2n​) 得知 k≥n2k\ge \frac n2k≥2n​ 还是 k<n2k\lt \frac n2k<2n​.

  1. 如果 k<n2k\lt \frac n2k<2n​，那么我们可以在全 000 的那半段上二分
  2. 如果 k≥n2k\ge \frac n2k≥2n​，那么我们可以在含 111 的半段加上全 000 的半段上二分

这两种情况需要二分的区间长度都是 n2\frac{n}{2}2n​ 的，所以总次数是 3+log⁡n−1≤323+\log n-1\le 323+logn−1≤32，所以可以通过。
